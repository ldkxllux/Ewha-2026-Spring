# 이화여자대학교 캡스톤디자인 기말 평가 스킬

## 개요

이 스킬은 캡스톤디자인 기말 평가를 위한 종합 가이드다. 평가는 크게 세 영역으로 나뉜다:

1. **발표 평가** (프레젠테이션 + 질의응답)
2. **프로젝트 보고서 평가** (PDF)
3. **GitHub 프로젝트 레포 평가**

세 영역이 서로 정합성(Alignment)을 갖추고 있는지도 핵심 평가 항목이다.

---

## 1. 발표 평가 루브릭

### 발표 구성 요소 (8개 섹션)

| 섹션 | 내용 | 핵심 체크포인트 |
|------|------|----------------|
| 1 | 문제의 포착, 동기 | 왜 이 문제인가? 실제 Pain Point가 보이는가? |
| 2 | 문제의 정의, 고객, 가치제안 | 고객 세그먼트가 명확한가? Value Proposition이 구체적인가? |
| 3 | 관련연구, 경쟁제품 | 경쟁 분석이 충분한가? 차별점이 드러나는가? |
| 4 | 제안 솔루션, 독특성 주장 | 독창성 주장이 논리적이고 명확한가? |
| 5 | 기술체계, 아키텍처, MVP 구성 | 기술 스택과 아키텍처가 설명되어 있는가? MVP가 실질적인가? |
| 6 | 라이브 데모 | 실제 동작하는 데모인가? 완성도는? |
| 7 | AI투명성 리포트 | AI 활용 내역이 투명하게 공개되어 있는가? |
| 8 | 질의응답 | 온라인 질문을 선택해 답변했는가? 심도 있는 답변인가? |

### 발표 시간 평가

- **권장**: 10분 발표 + 10분 질의응답
- **주의**: 너무 짧게 끝나는 것은 감점 요인
- **평가 포인트**: 주어진 시간을 충분히 활용했는가?

### 발표 평가 채점표 (총 40점)

| 항목 | 배점 | 세부 기준 |
|------|------|----------|
| 문제 정의 및 동기 | 5점 | Pain Point 명확성, 고객 정의, 가치제안 구체성 |
| 솔루션 & 독창성 | 8점 | 아이디어 참신성, 관련연구 대비 차별성 |
| 기술/아키텍처 | 7점 | 기술 체계 설명, MVP 완성도 |
| 라이브 데모 | 8점 | 실제 동작 여부, 완성도, 핵심 기능 시연 |
| AI투명성 리포트 | 5점 | AI 활용 내역 공개, 판단력 vs. 단순 의존 여부 |
| 발표 시간 활용 | 3점 | 10분 충분 활용, Q&A 활성화 |
| 질의응답 | 4점 | 답변의 깊이, 온라인 질문 선택 및 응답 방식 |

## 2. 프로젝트 보고서 평가 루브릭

### 파일명 규칙 (필수 체크 — 미준수 시 즉시 감점)

```
형식: [2자리팀번호]_[팀명]_...[기타].PDF
예시: 05_너네팀_캡스톤최종보고서.PDF
     12_우리팀_FinalReport.PDF
```

**체크 항목:**
- [ ] PDF 파일명 앞에 `2자리숫자 팀번호 + _` 로 시작하는가?
- [ ] 팀번호가 한 자리일 때 앞에 0이 붙어 있는가? (예: `05_` not `5_`)

### 커버 페이지 구성 요소

| 요소 | 위치/형식 | 필수 여부 |
|------|----------|----------|
| 트랙 표시 | 좌측 상단, 눈에 띄게 | ✅ 필수 |
| 팀번호 | 좌측 상단 | ✅ 필수 |
| 지도교수 성함 | 좌측 상단 | ✅ 필수 |
| 타이틀 (Self-Sufficient, Self-Explanatory) | 중앙, 굵고 크게 | ✅ 필수 |
| 브랜드명 (있다면) | 타이틀과 다른 줄 | 조건부 |
| 팀원 이름 & 학번 전체 | 본문 내 | ✅ 필수 |
| 날짜 | 커버 페이지 | ✅ 필수 |

> **Self-Sufficient & Self-Explanatory 타이틀**: 제목만 보고도 무슨 프로젝트인지 알 수 있어야 함. 모호한 제목은 감점.

### 보고서 내용 구성 체크리스트

```
필수 포함 섹션:
□ 문제의 포착
□ 문제의 정의
□ 고객 (Customer Segment)
□ 솔루션
□ 차별성 (독창성 주장)
□ Related Works (관련연구/경쟁제품)
□ 아키텍처
□ 실험 내용
□ 결과 분석
□ 마무리 (결론)
□ AI투명성 리포트
```

### 보고서 평가 채점표 (총 35점)

| 항목 | 배점 | 세부 기준 |
|------|------|----------|
| 파일명 규칙 준수 | 3점 | 팀번호_형식 준수 여부 |
| 커버 페이지 완성도 | 3점 | 트랙/팀번호/교수/타이틀/팀원/날짜 |
| 내용 완성도 (필수 섹션 포함) | 8점 | 모든 필수 섹션 포함 여부 |
| 논리적 구성 & 읽기 쉬운 정도 | 5점 | 구성의 흐름, 가독성 |
| 포맷팅 & 시각적 완성도 | 4점 | 표/그림/레이아웃의 완성도 |
| Project Brief와의 정합성 | 5점 | Brief 내용이 보고서에 잘 녹아있는가? |
| 기술 내용 & GitHub 레포 정합성 | 4점 | 보고서 기술 내용이 실제 코드와 일치하는가? |
| AI투명성 리포트 | 3점 | AI 활용 내역의 투명성 및 판단력 |

## 3. GitHub 프로젝트 레포 평가 루브릭

### Human Visitor-Friendly 체크리스트

```
□ README.md가 충실하게 작성되어 있는가?
□ 프로젝트 개요, 설치/실행 방법이 명확한가?
□ 기술 스택, 아키텍처 설명이 있는가?
□ 스크린샷 또는 데모 영상 링크가 있는가?
□ 팀 정보 (팀번호, 팀원, 지도교수)가 표시되어 있는가?
□ 라이선스 또는 기여 가이드가 있는가?
□ 커밋 히스토리가 팀 협업을 보여주는가?
```

### Project Brief와의 정합성 체크

- [ ] GitHub 레포의 내용이 Project Brief와 일치하는가?
- [ ] 레포 구조가 보고서에 서술된 아키텍처와 일치하는가?
- [ ] 발표 자료(슬라이드)가 레포에 포함 또는 링크되어 있는가?

### GitHub 레포 평가 채점표 (총 25점)

| 항목 | 배점 | 세부 기준 |
|------|------|----------|
| README 완성도 | 6점 | 개요/설치/실행/팀정보 포함 여부 |
| Human Visitor-Friendly 구성 | 5점 | 외부인이 봐도 이해 가능한가? |
| Project Brief 정합성 | 5점 | Brief 내용과 레포 내용 일치 여부 |
| 실제 코드/결과물 존재 | 5점 | 실제 작동하는 코드가 있는가? |
| 팀 협업 증거 | 4점 | 커밋 히스토리, 브랜치 활용 등 |

## 4. 종합 평가 절차

### Step 1: 기본 정보 확인
평가 전 다음 정보를 파악한다:
- 팀번호, 팀명, 트랙, 지도교수
- 제출 기한: **6월 22일(월) 23:59**
- GitHub 레포 URL
- PDF 보고서 파일명

### Step 2: 파일명 규칙 체크 (최우선)
> 사소해 보이지만 필수 요구사항 — 미준수 즉시 지적

### Step 3: 영역별 평가 진행
1. 발표 영상/슬라이드 검토
2. PDF 보고서 검토
3. GitHub 레포 검토

### Step 4: 정합성(Alignment) 검토
세 영역(발표 ↔ 보고서 ↔ 레포)이 일관된 내용을 담고 있는지 확인.

### Step 5: 종합 점수 산출

| 평가 영역 | 만점 |
|----------|------|
| 발표 | 40점 |
| 보고서 | 35점 |
| GitHub 레포 | 25점 |
| **합계** | **100점** |

## 5. AI투명성 리포트 평가 가이드

AI투명성 리포트는 발표와 보고서 **양쪽**에 포함되어야 한다.

**평가 포인트:**
- AI 도구 사용 내역이 구체적으로 기술되어 있는가?
- 단순히 AI 출력을 복붙했는가, 아니면 비판적 판단을 거쳤는가?
- AI를 "팀원"으로서 협업한 근거가 보이는가?
- 어느 부분에서 AI를 활용했고, 어느 부분은 스스로 판단했는지 명시되어 있는가?

**감점 사유:**
- AI투명성 리포트 자체가 없는 경우
- "ChatGPT 사용함" 수준의 형식적 기재만 있는 경우

## 7. 자주 있는 감점 패턴 (참고)

| 패턴 | 영역 | 영향 |
|------|------|------|
| 파일명에 팀번호 미포함 or 한 자리 숫자 | 보고서 | 즉시 -3점 |
| 발표가 7~8분에 끝남 | 발표 | 시간 활용 감점 |
| 라이브 데모가 없고 녹화 영상만 있음 | 발표 | 데모 완성도 감점 |
| AI투명성 리포트가 형식적임 | 발표/보고서 | 각 -2~3점 |
| README가 Default 상태임 | 레포 | Human Visitor-Friendly 감점 |
| 보고서와 레포 내용 불일치 | 정합성 | 다중 감점 |
| 커버 페이지에 팀원 학번 누락 | 보고서 | 커버 페이지 감점 |
| 타이틀이 모호함 (예: "AI 프로젝트") | 보고서 | 커버 페이지 감점 |
